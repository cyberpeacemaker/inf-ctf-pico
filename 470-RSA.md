# RSA Process
Spot on. In a real-world scenario, you are essentially moving between three "states": **Human Text** $\rightarrow$ **Raw Bytes** $\rightarrow$ **Large Integer**.

The RSA algorithm only knows how to do math on numbers, so we have to treat our data as one giant number. Here is the lifecycle of a message in a real-world RSA implementation:

---

### The RSA Workflow

| Step | Action | Format | Process |
| :--- | :--- | :--- | :--- |
| **1** | **Prepare** | String | You start with a message like `"Hello!"`. |
| **2** | **Encode** | Bytes | You turn the string into bytes (UTF-8). |
| **3** | **Pad** | Bytes | **Crucial Step:** You add random data (Padding) to make it secure. |
| **4** | **Convert** | Integer | You treat those bytes as one massive number ($M$). |
| **5** | **Encrypt** | Integer | You calculate $C = M^e \pmod{N}$. |
| **6** | **Decrypt** | Integer | The receiver calculates $M = C^d \pmod{N}$. |
| **7** | **Unpad/Decode**| String | Remove padding and convert bytes back to `"Hello!"`. |


# Factor Tip

#### Step 1: The "Factordb" Check
The first thing every CTF player does with a modulus $N$ is check [factordb.com](https://factordb.com). This is a massive database of known factorizations.
* **Result for your $N$:** If you paste that number into Factordb, you will likely see it is already "Fully Factored."

#### Step 2: Look at the last digit
Your $N$ ends in **6**.
* This is a **massive red flag**.
* Primes (except for 2) always end in 1, 3, 7, or 9.
* If $N$ is even (ends in 0, 2, 4, 6, or 8), it means **one of the factors is 2**.
* If one "prime" is 2, the other "prime" is simply $N / 2$.

Wait—if the "primes" are $2$ and some other number, the encryption is completely broken because $p=2$ is not a large prime at all.
